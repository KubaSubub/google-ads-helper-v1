BUILD — implementacja task-by-task
main ctx auto-test hook

Kazdy krok to osobny Task. Po kazdej edycji pliku hook post-edit-test.sh automatycznie odpala pytest lub vite build. Nie idziesz dalej jesli test FAIL.

  • 1
    Backend model/schema (jesli nowe)
  • 2
    Backend service (logika biznesowa)
  • 3
    Backend router/endpoint
  • 4
    Backend tests
  • 5
    Frontend component
  • 6
    Frontend integration (import, routing)
Hook: PostToolUse → po kazdym Edit/Write → auto-test → FAIL = napraw przed kolejnym taskiem
Git checkpoint: Po zakonczeniu BUILD → git commit -m "wip: phase-2 {task}"
Checkpoint do rollbacku jesli VERIFY/TEST fail. Squashowany w FAZIE 6 (SHIP) przez git reset --soft → czysty commit.
3
VERIFY — 3 review agenty rownolegle
fork 3x parallel gate 7/10

3 niezalezne agenty w izolowanych kontekstach. Kazdy ma read-only tools (Read, Grep, Glob). Wyniki sa mergowane w jeden raport. Srednia musi byc >= 7/10.

Code Quality
code-quality-reviewer • Sonnet
  • Pattern matching z codebase
  • DRY / SOLID
  • Naming conventions
  • Error handling
  • Dead code
Security
security-reviewer • Sonnet
  • SQL Injection (ORM vs raw)
  • XSS / CSRF
  • Auth bypass
  • Secrets in code
  • OWASP Top 10
Domain Expert
google-ads-domain • Sonnet
  • Micros handling (cost/bid)
  • Conversions = float
  • Circuit breaker on writes
  • Keyring storage
  • Playbook alignment
Gate: Srednia 3 ocen >= 7/10 → PASS. Jesli FAIL → wroc do FAZA 2, napraw findings. Max 3 iteracje, potem eskalacja do usera.
4
TEST — pelny test suite
2x parallel gate pass/fail

Backend i frontend testowane rownolegle. Oba musza przejsc.

Backend
pytest • 503+ testow
  • cd backend && pytest
  • Pelny suite z fixtures
  • Coverage report
Frontend
vite build + opcjonalnie Playwright
  • npm run build
  • Compile/type errors
  • Playwright screenshots (opcja)
Gate: Oba PASS → dalej. Ktorykolwiek FAIL → wroc do FAZA 2. Max 3 iteracje.
5

Tylko dla zmian w UI/frontend. Dwa etapy: symulacja PPC specjalisty, potem ocena eksperta. Jesli ocena < 7/10 → generuj plan naprawczy i wroc do BUILD.

SHIP — /done pipeline
main ctx docs fork pm gate 7/10

Commit, dokumentacja (w izolacji), PM gate, push. Jesli PM < 7/10 → wroc do BUILD.

  • 0
    Squash WIP commits — git reset --soft HEAD~N → wszystkie zmiany staged
  • 1
    /commit — inteligentny commit z conventional prefix (1 czysty commit zamiast WIP)
  • 2
    /docs-sync fork — aktualizacja PROGRESS.md, API_ENDPOINTS.md
  • 3
    git commit docs — osobny commit na dokumentacje
  • 4
    /pm-check fork — ocena PM >= 7/10 → tworzy marker
  • 5
    git push — pre-push hook sprawdza marker PM
Gate: PM score >= 7/10 → push. Jesli < 7 → lista problemow, wroc do FAZA 2. Max 2 iteracje.
